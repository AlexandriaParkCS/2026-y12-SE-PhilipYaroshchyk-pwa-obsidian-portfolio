BreakingBad app code review

**Strengths:**
1. Routes live in src/app.py, business logic in src/user_service.py, and DB operations in src/sqldb.py. That’s a good foundation.
2. Parameterised SQL is used for inserts/selects (e.g., WHERE username = ?), which is the correct anti–SQL-injection pattern.
3. CSRF tokens are present in HTML forms (csrf_token() hidden input) and CSRFProtect(app) is enabled in the Flask app.
4. CSP is being considered (both as a header and as a meta tag), an attempt is made to use a nonce for scripts in summary.html. That’s advanced thinking for a beginner PWA.

**Issues Found:**
*High priority fixes:*
- Password hashing is implemented incorrectly (and insecurely). In src/user_service.py:
- A single fixed salt is hard-coded: self.salt = b'$2b$12$pXlzeoRw6XKXNyBbufFrYu'
- Login recomputes the hash and compares equality:
    - password_hash = bcrypt.hashpw(password.encode('utf-8'), self.salt)
    - if user["password_hash"] == password_hash:
*Issues:*
- Using a fixed salt defeats a major purpose of bcrypt and makes password cracking easier.
- Best practice is to store the bcrypt hash (which already contains its salt), and verify with bcrypt.checkpw(plain, stored_hash).
- Some routes are not secure because they assume the user is logged in but do not check or enforce that assumption. Example: @app.route('/summary/<goal_id>') calls: user_service.get_transaction_summary_for_a_goal(session['user_id'], goal_id)
- But there is no guard like if 'user_id' not in session: return redirect(...). If someone hits this endpoint without a session, it will throw a KeyError and may leak error behaviour.
- Also, routes like /add_expense and /add_goal use session["user_id"] directly without a login check. Add consistent access control (a simple login_required decorator is a great learning exercise).

*DB schema improvements:*

- transactions.transaction_date, goals.start_date, goals.end_date are stored as TEXT. For SQLite this can work, but then you must enforce consistent YYYY-MM-DD and be careful with comparisons. Consider using ISO dates consistently and validating server-side.
- Amount stored as FLOAT. For money, floats can introduce rounding errors. A common improvement is to store the amount in cents (amount_cents INTEGER) or use DECIMAL patterns (SQLite doesn’t strictly enforce decimal precision, so storing the amount in cents is simplest).
- The DB initialisation occurs on import. In src/app.py: sql_db = SqlDb("../runtime/db/sql.db") runs at import time and triggers table creation. This works for a small app, but it can cause trouble later (tests, scaling, multiple workers). Better is: create app + db in a factory/init step.


*Client-side patterns are easy to bypass.*

- Good HTML pattern= attributes (email, password complexity, date format). But those are easy to bypass. Server-side must validate too:
- amount = float(request.form["amount"]) in /add_expense can throw exceptions or accept weird formats.
- Dates are accepted as strings and passed straight to DB. Best practice is: “Validate on the server; treat the browser as untrusted.”
- Error handling is inconsistent. Some routes catch Exception and show a friendly message; others will crash if form/session data is missing.

- Also, logging currently mixes: print(...) statements, and log.info(...) with a file logger. Use the logger everywhere and avoid printing sensitive values.

**Recommendations:**
*Must fix:*

- Fix bcrypt usage: per-user salt + checkpw.
- Enforce login checks on every authenticated route (/add_expense, /add_goal, /summary/<goal_id>, etc.).
- Add server-side validation for amount/date/strings.
- Replace prints with structured logging; don’t log sensitive data.
- Fix naming/typos (Transation) and stop shadowing list.
- Consider integer cents for money.
- Convert model objects to dataclasses for readability.
